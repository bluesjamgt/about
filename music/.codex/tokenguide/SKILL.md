# Tokenguide

## 目標

在編輯 Blues Jam GT 音樂頁面時，讓每個開發回合保持範圍明確、可驗證，並降低 Token 消耗。

1. 只讀取目前需求所需的檔案與行號範圍。
2. 優先使用精準搜尋結果，不進行整份檔案閱讀。
3. 避免掃描不相關資料夾或舊功能區域。
4. 可行時，將同一檔案的相關修改合併為一次 patch。
5. 最後只回報：已修改檔案、變更摘要與驗證結果。

## 任務規模

讀取檔案前，先判定任務規模。

* 小型 CSS 或文字修改：讀取對應 selector 或附近行數，直接修正，再做 selector 層級確認。
* 中型 UI 互動修改：各讀取一次相關 HTML 結構、JavaScript 函式或狀態區塊，以及 CSS selector 區塊。
* 大型重構：先建立簡短計畫，再進行聚焦測試。

## 讀取規則

1. 先使用 `Select-String` 或 `rg` 搜尋函式名稱、狀態欄位、DOM selector 與 CSS selector。
2. 若需再次讀取同一檔案，應從相鄰行數範圍延續閱讀，不要重新從檔案開頭讀取。
3. 已確認不相關的檔案，不要再次開啟。
4. 若先前上下文已確認檔案位置，直接前往已知區域：

   * FORM 結構：`index.html`
   * 圖層狀態、render 與 toggle 邏輯：`assets/modules/axon/app.js`
   * 底部 dock 與快捷 UI：`assets/styles/workbench.css`
5. 純視覺調整時，除非 selector 或 DOM 結構未知，否則不要搜尋 HTML 與 JavaScript。

## 編輯規則

1. 同一檔案內的多項 UI 調整，盡量合併為一次 patch。
2. DOM、狀態與 CSS 的修改，應依責任範圍分組。
3. 修正版面、顏色、標籤或動畫時序時，不得順便重構不相關程式碼。
4. 除非使用者明確要求，不得碰觸匯出、palette 持久化、transpose 動畫或其他不相關功能。

## 驗證規則

採用符合修改範圍的最小驗證迴圈。

* JavaScript 修改：對修改過的 JS 檔案執行 `node --check`。
* 僅修改 CSS：確認變更過的 selector，或進行輕量 CSS 語法／selector 檢查；不要執行 JS 檢查。
* 瀏覽器檢查：只測試核心互動，最多再加一個邊界案例。
* 已驗證且未被修改的區域，不必重新測試。
* 若無法進行視覺驗證，最後必須明確說明，然後停止。

## 溝通規則

1. 進度更新保持簡短，尤其是小型 UI 修改。
2. 不要解釋每個微小發現。
3. 除非會影響結果，否則不要重複說明工具限制。
4. 最後回覆應包含：

   * 已修改檔案
   * 變更摘要
   * 驗證結果

## 常見浪費，應避免

1. 只調整 CSS 位置，卻重讀 HTML、JS 與 CSS。
2. 每次微小 patch 後，都重複執行多輪 `Select-String`。
3. 對沒有修改的 JPG 匯出、palette 儲存或 transpose 動畫重複驗證。
4. 把視覺微調當成完整功能測試。
5. 因為上一回合很複雜，就在這一回合無限制擴張上下文，即使目前任務很簡單。

## 優先原則

正確性、資安、資料完整性與可回復性，優先於節省 Token。

若修改可能影響共用狀態、建置產物、資料格式、跨頁行為、共用元件、依賴套件或安全性，必須擴大讀取範圍與驗證，不得為了節省 Token 而省略必要檢查。
